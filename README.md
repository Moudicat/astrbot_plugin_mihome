# 🌟 AstrBot MiHome Control (v6.4.0 )

基于 [AstrBot](https://github.com/Soulter/AstrBot) 框架开发的米家智能家居云端控制插件。

旨在为用户提供**最安全、最直观**的米家设备控制体验。

## ✨ 核心特性

* **🔒 隔离式扫码登录**：登录流程通过独立子进程完成，避免阻塞主事件循环；认证凭证安全保存在 `plugin_data` 目录中，便于备份、迁移与卸载清理。
* **🧠 设备画像驱动 (Device Profiles)**：将用户别名、中文控制词与底层属性键解耦。插件内置常见设备的专属画像，可将“睡眠”“制冷”“童锁”等中文指令映射为对应设备可识别的底层属性和值。
* **📊 中文状态展示**：基于本地画像与云端返回结果，对设备状态项进行中文翻译与单位美化展示。像 `relative_humidity`、`pm2.5_density` 这类底层字段会尽量转换为“当前湿度: 60%”“PM2.5浓度: 12”这类更易读的文本。
* **🛡️ 柔性降级与防呆设计**：当设备未配置别名、未同步缓存、暂时离线、深度休眠或部分状态读取失败时，插件会优先返回本地设备画像与引导信息，尽可能保证“可看、可控、可排障”。
* **🔑 基础控制与高级属性控制**：对所有设备均已适配直接开关，对于所有设备均支持通过原始属性调整值，例如直接下发temperature 26来调整为26度，其中已测试的设备已适配中文。

## 🚀 快速上手：从零到控制你的第一台设备

对于初次使用的用户，请严格按照以下 5 步完成设备绑定与控制：

### 步骤 1：扫码授权登录
对机器人发送 `/米家登录`。  
机器人会拉起独立登录进程并返回一个授权链接，点击打开后使用米家 APP 扫码完成授权。授权成功后，凭证将安全保存在本地。

---

### 步骤 2：获取设备 DID 与云端名称（⚠️ 关键）
发送 `/刷新米家`。机器人会从云端拉取您账号下可见的设备列表，例如：
> `1. 【未配置别名】(米家智能空气炸锅 5.5L 可视版) [🟢] (1234567)`  
> `2. 【未配置别名】(我的锅) [🟢] (1234567)`  
> `3. 【未配置别名】(卧室空调) [🟢] (12345675677)`

其中：
- `【未配置别名】`：表示你还没在插件里给它取控制用别名
- 括号里的内容，例如 `米家智能空气炸锅 5.5L 可视版` / `我的锅` / `卧室空调`：是米家云端当前显示名称
- 行尾括号中的数字，例如 `1234567`：就是设备 DID

这一步非常关键。  
插件会在后台缓存设备的基础档案（如 DID、云端名称等），后续 `/米家详情`、`/米家帮助`、`/米家控制` 都依赖这份本地档案。  
请记录下你要控制设备对应的 DID。

---

### 步骤 3：在 WebUI 配置好记的设备别名
*⚠️ 出于安全性与精确性考虑，未配置别名的设备默认不允许查看详情与远程控制。*
前往 AstrBot WebUI 的插件配置页，找到 `device_map`，为设备 DID 配置一个你习惯使用的别名：

```json
{
  "卧室净化器": "1234567",
  "客厅 空调": "1234567",
  "空气炸锅": "4567567"
}
```
**说明：**
* **键（Key）**：你希望在机器人里使用的设备别名（支持带空格别名，例如："客厅 空调"）
* **值（Value）**：`/刷新米家` 输出中的设备 DID

---

### 步骤 4：为设备指定类别（⚠️ 强烈推荐配置）
接着在插件配置页填写 `device_category_map`，用于告诉插件“这台设备属于哪一类”。例如：

```json
{
  "卧室净化器": "净化器类别",
  "客厅 空调": "空调类别",
  "空气炸锅": "空气炸锅类别"
}
```
目前支持的类别有：
* `空调类别`
* `净化器类别`
* `风扇类别`
* `蒸煮锅类别`
* `空气炸锅类别`
* `无类别`

**说明：**
* **键（Key）**必须与你 `device_map` 中配置的别名完全一致
* **值（Value）**必须是上述合法类别之一
* 如果填写了非法类别，系统会自动按 `无类别` 处理

**为什么这一步重要？**
因为米家云端名称可能是用户自己改过的，例如：`卧室空调`、`我的锅`、`厨房冰箱`。这些名字虽然适合展示，但不适合直接用于中文控制模板匹配。
配置设备类别后，插件就能更准确地输出：
* 中文化的可调属性
* 中文化的状态传感项
* 更友好的 `/米家帮助`
* 更稳定的 `/米家详情`

填写后保存配置，重载插件，然后再执行一次 `/刷新米家`，让缓存与映射同步生效。

---

### 步骤 5：开始查看与控制
此时你可以先发送详情指令，查看设备能力与当前状态：
> `/米家详情 卧室净化器`

如需查看常用控制写法：
> `/米家帮助 卧室净化器`

再发送控制指令：
> `/米家控制 卧室净化器 开`
> `/米家控制 卧室净化器 模式 睡眠`

如果设备成功响应，说明你的智能管家已经可以正常工作了。

## 🎮 指令指南

### 🔑 账号与管理
* `/米家登录`：拉起独立登录流程，获取扫码授权链接。
* `/米家登出`：清除当前登录状态与本地凭证。
* `/米家状态`：查看插件当前登录状态、共享设备异常与最近一次控制记录。

### 📡 设备查询
* `/刷新米家`：同步云端设备列表，输出带 DID 的设备总览，并刷新本地设备档案缓存。
* `/米家详情 [设备别名]`：查看该设备详情。
  * 若已配置有效类别：优先返回中文化的本地设备画像（可调属性、状态传感），再尝试补充实时状态数据。
  * 若类别为 `无类别`：直接进入原始能力探测，返回设备支持的原始英文属性菜单。
* `/米家帮助 [设备别名]`：查看该设备常用控制方式、支持的控制属性与推荐指令示例。

### 🎛️ 设备控制

**基础电源控制（推荐优先使用）**
* `/米家控制 [设备别名] [开/关]`
  * 示例：`/米家控制 落地扇 开`

**高级属性控制**
* `/米家控制 [设备别名] [属性] [值]`
  * 示例：`/米家控制 空调 温度 26`
  * 示例：`/米家控制 蒸煮锅 火力 3`

> 💡 **提示：**
> 1. 推荐优先参考 `/米家帮助 [设备别名]` 中给出的示例格式。
> 2. 某些设备支持的高级属性较多，且不同型号之间取值方式可能不同（数字、布尔、枚举混用）。
> 3. 对于尚未完全适配的新设备，若 `/米家详情` 已展示原生英文属性名，你仍可尝试使用这些原始属性名进行高级控制透传。

## 🧠 关于“设备类别”与“兜底能力探测”

本插件采用“用户手动指定设备类别 + 原始能力探测兜底”的设计：

**如果设备已配置有效类别**，例如 `{"卧室净化器": "净化器类别"}`，那么 `/米家详情 卧室净化器` 会优先返回类似：
```text
📖 【卧室净化器】本地设备画像:
✅ 可调属性: 屏幕, 开关, 提示音, 模式, 童锁, 风速
📡 状态传感: PM2.5浓度, 当前温度, 当前湿度, 滤芯剩余天数, 滤芯寿命百分比
```
然后再尝试读取实时值。

**如果设备未配置类别，或类别为 `无类别`**，例如 `{"手环": "无类别"}`，那么 `/米家详情 手环` 会进入能力探测模式，返回类似：
```text
⏳ 正在探测【手环】的能力菜单...
✅ 【手环】支持的高级属性:
battery_level, steps, heart_rate, ...
```
这意味着即使当前设备还没有中文模板，插件依然会尽量把它支持的原始属性列出来，方便你自行摸索控制方式。

**如果设备已配置类别，但实时数据读取失败**，插件会尽量返回“已知状态项”或原始能力菜单，例如：
```text
⏳ 正在向米家云端精准读取实时数据，请稍候...
📡 已知状态项 (当前实况获取失败或无数据):
ac_state, ac_type, brand_id, electric_power, match_state, remote_id
```
这样即使当前实况没读出来，你也仍然可以知道设备理论上暴露了哪些属性。

1. 详情页存在大量未汉化字段
2. 控制属性映射不准确
3. 某型号设备的值域和文案不匹配
4. 同一设备类别下还有额外原始属性未纳入模板

欢迎提交 Issue，帮助完善设备类别模板与控制映射。建议附带以下信息：
* 设备的米家云端名称
* 你在插件配置中的设备别名
* 你为该设备配置的类别
* `/米家详情 [设备别名]` 的返回内容
* `/米家帮助 [设备别名]` 的返回内容
* 实际可用但当前未适配的控制方式

例如可直接贴出：
```text
✅ 【蒸煮锅】支持的高级属性:
status, mode, left_time, fault, heat_level, target_temperature, temperature, target_time, recipe_command, alarm, power_set, temp_reached, cook_done, temp_reached_set, lifetime, recipeid, recipe_sort, mode_sort, recipes_recipe_command, cookreservation, keepwarm_set, stepid, warm_temperature, user_device_info, recipename, switch_pausetoadd
```

## ⚠️ 常见排障

| 现象 | 可能原因与建议 |
| :--- | :--- |
| **设备找不到 / 共享设备异常** | 第三方接口对共享设备的支持并不总是稳定。建议优先使用设备**主账号**扫码登录。 |
| **设备拒绝请求** | 你可能修改了只读属性，或传入了不符合设备要求的值类型/值范围。建议先查看 `/米家帮助 [设备别名]`。 |
| **提示需先刷新米家** | 本地缓存中还没有该设备的基础档案。请先执行一次 `/刷新米家`。 |
| **详情页只有能力菜单，没有中文画像** | 该设备未配置有效类别，或当前类别被识别为 `无类别`。请检查 `device_category_map`。 |
| **详情页只有本地画像，没有实时状态** | 设备可能离线、深度休眠，或者当前接口未成功读取状态。这种情况下通常仍可尝试发送基础控制指令。 |
| **云端名称和别名不一致** | 这是正常现象。别名是你在插件里配置的控制入口名；云端名称是米家 App 中当前显示的设备名称。 |
| **网络异常 / 超时** | 宿主机与米家接口之间可能发生 TLS 断连、网络波动或请求受限。建议稍后重试，并检查网络环境。 |

* **原生/本地场景支持**：支持调用米家 App 预设场景或在插件内自定义快捷指令组（如 `/米家场景 晚安`）。
