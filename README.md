# 🌟 AstrBot MiHome Control

基于 [AstrBot](https://github.com/Soulter/AstrBot) 框架开发的支持自然语言调用的米家智能家居云端控制插件。

<div align="center">

[![Moe Counter](https://count.getloli.com/get/@astrbot_plugin_mihome?theme=moebooru)](https://github.com/674537331/astrbot_plugin_mihome)

</div>

---

## ✨ 核心特性

* 🔒 **扫码登录**：认证凭证安全保存在 `plugin_data` 目录中，便于备份、迁移与卸载清理。
* 🚀 **场景调用**：直接通过米家APP内已有场景对多个设备进行复杂调整，支持指令和LLM工具自然语言调用！
* 🧠 **设备画像**：将用户别名、中文控制词与底层属性键解耦。
* 📊 **中文状态**：基于本地画像与云端返回结果，对设备状态项进行中文翻译与单位美化展示。
* 🔑 **设备兼容**：对所有设备均已适配直接开关，并支持通过原始属性（如 `temperature 26`）进行透传控制。

## 📖 使用指南

<details>
<summary><b>🚀 快速上手：从零到控制你的第一台设备 (点击展开)</b></summary>

对于初次使用的用户，请严格按照以下 5 步完成设备绑定与控制：

### 步骤 1：扫码授权登录
对机器人发送 `/米家登录`。机器人会返回一个授权链接，打开后使用米家 APP 扫码完成授权。

### 步骤 2：获取设备 DID（关键）
发送 `/刷新米家`。机器人会从云端拉取您账号下可见的设备列表，例如：

```text
1. 【未配置别名】(米家智能空气炸锅 5.5L 可视版) [🟢]
   DID: 1234567
   model: careli.fryer.maf07c

2. 【未配置别名】(主卧 体脂秤) [🟢]
   DID: 7654321
   model: yunmai.scales.ms103
```

其中：
- `【未配置别名】`：表示你还没在插件里给它取控制用别名
- 括号里的内容，例如 `米家智能空气炸锅 5.5L 可视版` / `主卧 体脂秤` / `卧室空调`：是米家云端当前显示名称
- `DID`：设备 DID，**用于填写 `device_map`**
- `model`：设备型号，**用于设备画像适配、提交 Issue、排查兼容性问题**

请记录下你要控制设备对应的 **DID**。  
如需反馈适配问题，也建议同时记录对应的 **model**。

### 步骤 3：配置设备别名
前往 AstrBot WebUI 插件配置页，找到 `device_map`，为 DID 配置一个别名：
```json
{
  "卧室净化器": "1234567",
  "客厅 空调": "8888888"
}
```

### 步骤 4：指定设备类别（推荐，若设备已在上表“完美适配”，则可忽略）
接着在插件配置页填写 `device_category_map`，用于告诉插件“这台设备属于哪一类”。例如：

```json
{
  "卧室净化器": "净化器类别",
  "客厅 空调": "空调类别",
  "空气炸锅": "空气炸锅类别",
  "扫地机": "扫地机类别",
  "厨房 热水壶": "热水器类别",
  "书房 路由器": "路由器类别"
}
```
目前支持的类别有：
* `空调类别`
* `净化器类别`
* `风扇类别`
* `蒸煮锅类别`
* `空气炸锅类别`
* `体脂秤类别`
* `温湿度计类别`
* `扫地机类别`
* `热水器类别`
* `路由器类别`
* `无类别`

说明：
* 键（Key）必须与你 `device_map` 中配置的别名完全一致
* 值（Value）必须是上述合法类别之一
* 如果填写了非法类别，系统会自动按 `无类别` 处理

为什么这一步重要？
因为米家云端名称可能是用户自己改过的。配置设备类别后，插件就能赋予它：
* 中文化的可调属性
* 中文化的状态传感项
* 更友好的 `/米家帮助`
* 更稳定的 `/米家详情`

💡 特别说明（模型降级机制）：
本插件内置了型号精确匹配 机制。如果你设备的具体型号（Model，如 `yunmai.scales.ms103`）已经被开发者直接写进了核心字典库中，插件将无视你在这里配置的类别，直接采用最完美的型号专属配置。配置类别主要是为了给尚未收录专属型号的设备进行泛用性兜底。

填写后保存配置，重载插件，然后再执行一次 `/刷新米家`，让缓存与映射同步生效。

### 步骤 5：开始测试
此时你可以先发送详情指令，查看设备能力与当前状态：
> `/米家详情 卧室净化器`
> `/米家详情 扫地机`
> `/米家详情 厨房 热水壶`

如需查看常用控制写法：
> `/米家帮助 扫地机`
> `/米家帮助 厨房 热水壶`

再发送控制指令：
> `/米家控制 卧室净化器 开`
> `/米家控制 扫地机 开始清扫`
> `/米家控制 厨房 热水壶 温度 85`

* `/米家详情 [设备别名]`：查看该设备详情。

高级属性控制
* `/米家控制 [设备别名] [属性] [值]`
  * 示例：`/米家控制 空调 温度 26`

动作触发控制
* `/米家控制 [设备别名] [动作名]`
  * 示例（中文）：`/米家控制 扫地机 开始清扫`
  * 示例（英文）：`/米家控制 扫地机 start_sweep`
  * 示例（手误兼容）：`/米家控制 扫地机 start-sweep` 或 `start sweep`

> 💡 提示：
> 1. 推荐优先参考 `/米家帮助 [设备别名]` 中给出的示例格式。对于休眠类设备（如体脂秤），请在屏幕亮起时发送指令。
> 2. 对于尚未完全适配的新设备，若 `/米家详情` 已展示原生英文属性名，你仍可尝试使用这些原始属性名进行高级控制透传。

<summary><b>🎬 米家智能场景支持（Scenes）（推荐的使用方式） (点击展开)</b></summary>

支持执行米家 APP 中已配置的“手动执行”场景。

### 📱 前置准备
1. 打开米家 APP，进入底部的 “智能” 页面。
2. 点击右上角 “+” 创建新智能。
3. 触发条件请选择：“手动执行”。
4. 执行动作中添加你希望联动的一系列设备操作（如：关灯、关空调、关闭净化器、拉窗帘等）。
5. 保存并起一个有辨识度的名字，例如：
   - `晚安模式`
   - `离家全关`
   - `关净化器`
   - `影院模式`
> 建议尽量避免过于模糊或重复的场景名，例如仅命名为 `晚安`、`离家`、`开灯`。  
> 更清晰的命名有助于指令匹配，也更利于大模型安全调用。

### 🎮 场景控制指令

#### `/米家场景列表`
同步并查看当前账号下所有可执行的云端场景。
功能说明：
- 插件会从米家云端拉取当前账号下的场景列表
- 成功后会将结果缓存到插件本地，供后续大模型只读 Tool 使用
- 输出内容通常包括：
  - 场景名称
  - `scene_id`
  - 家庭标识（若可获得）

示例：
`/米家场景列表`

#### `/米家场景 [场景名或scene_id]`
立即执行指定场景。

示例：
`/米家场景 晚安模式`  
`/米家场景 123456789`

#### ⚠️ 同名场景保护
如果你的米家账号下存在多个同名场景，插件为了避免误操作，不会直接执行，而是提示你：
- 改用唯一的 `scene_id`
- 或前往米家 APP 修改为更容易区分的名称

因此，如果你：
- 新增了场景
- 修改了场景名称
- 删除了旧场景

`/米家场景列表`

- 用户明确提出家居控制需求，但模型不确定应该执行哪个场景
- 模型需要先确认可用场景名称

#### 2. 执行 Tool：执行场景
大模型可以执行某个已缓存并可匹配到的场景。

- `执行晚安场景`

### 🛡️ 安全边界
本插件支持为大模型提供 **米家设备只读查看 Tool**，但该能力被严格限制为“只读”。

**大模型只允许：**
- 查看已配置别名设备的实时状态
- 读取温湿度、空气质量、电源状态、运行模式等只读信息

**大模型不允许：**
- 打开或关闭设备
- 修改任意设备属性
- 执行动作
- 透传 `/米家控制`
- 访问未配置别名的设备

也就是说，这个 Tool 只能“看”，不能“动”。

---

### ⚙️ 如何启用
1. 打开 AstrBot WebUI 的插件配置页面  
2. 找到并开启：`enable_readonly_tool`  
3. 保存配置并重载插件  

---

### 📌 重要规则：只能从别名中检索
大模型不能自由扫描你米家账号下的全部设备。  
它只能从你在 `device_map` 中**已经手动配置过的设备别名**里检索目标设备。

例如你配置了：
```json
{
  "卧室 温湿度计": "123456789",
  "卧室 净化器": "987654321",
  "客厅 空调": "888888888",
  "厨房 热水壶": "666666666",
  "书房 路由器": "777777777"
}
```

那么大模型只允许读取：
- 卧室 温湿度计
- 卧室 净化器
- 客厅 空调

而**不能**读取未配置别名的其他设备。

#### 📂 建议同时配置设备类别
为了让只读 Tool 更稳定地返回中文化状态，建议你在 `device_category_map` 中为相关设备配置类别，例如：

```json
{
  "卧室 温湿度计": "温湿度计类别",
  "卧室 净化器": "净化器类别",
  "客厅 空调": "空调类别",
  "厨房 热水壶": "热水器类别",
  "书房 路由器": "路由器类别"
}
```

**说明：**
- 如果设备型号已经命中插件内置的 `MODEL_PROFILES`，即使不手填类别，通常也能正常识别。
- 如果设备型号未命中内置模板，但你又想让大模型读取它，建议手动填写类别。

---

### 🧠 大模型可用的两个只读 Tool

**1. `list_configured_mihome_aliases`**
列出当前插件中已经配置好的米家设备别名。
*适用场景：*
- 模型不确定你有哪些设备别名。
- 用户说了“卧室那个温湿度计”“净化器”“空调”这类目标，但模型需要先确认精确别名。

**2. `read_mihome_device_status_by_alias`**
按精确别名实时读取设备当前状态。
*适用场景：*
- 用户明确要“看状态”。
- 用户询问温湿度、空气质量、电源状态、工作模式等信息。
- 模型已经确认了要读取的设备别名。

---

### ⏱️ 不使用缓存，默认实时读取
这个只读 Tool 不会使用状态缓存。每次调用时，都会尽量直接读取当前设备的实时状态。

这样做的目的是避免以下误解：
- 用户问的是“现在”的状态
- 机器人却返回几分钟前的旧数据

**因此：**
- 设备目录 / 别名映射可以缓存
- 设备实时状态默认不缓存

---

### ✅ 适合触发只读 Tool 的自然语言
以下表达通常适合触发米家设备只读 Tool：
- `帮我看看卧室的温湿度`
- `现在净化器开着吗`
- `帮我看下客厅空调现在什么模式`
- `看看现在空气质量怎么样`
- `卧室温度适不适合睡觉`

### ❌ 不适合触发只读 Tool 的情况
以下情况通常不应调用只读 Tool：
- 纯闲聊、寒暄
- 没有明确设备目标，也没有明确状态查询意图
- 想要直接控制设备的请求（这应属于控制或场景，而不是只读查看）

例如：
- `晚安`
- `我要睡了`
- `帮我开空调`
- `把净化器关了`

这些并不是“只读查看”请求。

### 💬 交互示例

**用户自然语言：**
> `@机器人 现在帮我看看卧室的温湿度，适不适合直接睡觉？`

**理想流程：**
1. 大模型先确认当前已配置别名。
2. 找到类似 `卧室 温湿度计` 的设备别名。
3. 调用只读 Tool 读取实时数据。
4. 再结合读数进行自然语言解释。

*例如可能返回类似结论：*
> 当前温度略高，湿度稍偏高。可以睡，但不算特别舒适。如果你对睡眠环境敏感，可以考虑降低温度或改善通风。

### 🔄 推荐使用习惯
为了获得更稳定的体验，建议遵循以下流程：
1. 先通过 `/刷新米家` 获取设备 DID。
2. 在 `device_map` 中配置清晰的中文别名。
3. 尽量在 `device_category_map` 中为相关设备配置类别。
4. 开启 `enable_readonly_tool`。
5. 再让大模型进行只读状态查询。

</details>

## 🔬 进阶与反馈

<details>
<summary><b>🧠 技术细节：三层降级解析机制 (点击展开)</b></summary>

本插件采用以下顺序解析设备能力：
1. 型号精准匹配：若设备型号命中内置 `MODEL_PROFILES`，展现最完美的专属翻译。
2. 大类模板兜底：若未命中型号但配置了类别，使用通用中文模板。
3. 原始能力探测：若为 `无类别`，返回原始英文属性菜单，支持英文指令透传。

</details>

<details>
<summary><b>💡 设备画像未完全适配？ (点击展开)</b></summary>

欢迎提交 Issue 帮助我们完善核心字典库（MODEL_PROFILES）。  
请附带 `/astrbot/data/plugin_data/astrbot_plugin_mihome` 下对应的 `.json` 文件，或在终端执行以下代码并提供输出：

```bash
python - <<'PY'
from mijiaAPI import get_device_info
import json
model = "你的设备model（注意：不是 /刷新米家 中显示的 DID）"
info = get_device_info(model)
print(json.dumps(info, ensure_ascii=False, indent=2))
PY
```

说明：
- `/刷新米家` 当前会同时展示设备 `DID` 与 `model`
- `device_map` 里填写的是 `DID`
- `get_device_info()` 这里填写的必须是 **model**，不是 DID
- 如果暂时拿不到 model，也可以先附上插件缓存目录下对应的 `.json` 文件协助排查
</details>

## ⚠️ 常见排障与常见问题

| 现象 | 可能原因与建议 |
| :--- | :--- |
| 设备找不到 / 共享异常 | 建议优先使用设备主账号扫码登录。 |
| 设备拒绝请求 | 可能修改了只读属性或值越界。请参考 `/米家帮助`。 |
| 详情页只有英文菜单 | 设备未内置且未配置类别。请检查 `device_category_map`。 |
| 提示实况获取失败 | 设备可能离线或处于深度休眠（如体脂秤未踩亮）。 |
| 网络异常 / 超时 | 可能发生 TLS 断连或被防火墙风控，建议稍后重试。 |
