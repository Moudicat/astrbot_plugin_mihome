## [v6.4.0 ] - 2026-03-13

解决了米家云端设备已经被用户改名后，详情页画像失效、帮助不准确、未适配设备信息量不足等问题。

#### ✨ 新增
- 新增插件配置项 `device_category_map`
  - 采用“别名 -> 类别”的方式手动指定设备大类
  - 当前支持：
    - `空调类别`
    - `净化器类别`
    - `风扇类别`
    - `蒸煮锅类别`
    - `空气炸锅类别`
    - `无类别`
- 新增“原始能力菜单探测”兜底机制
  - 当设备未配置类别，或类别为 `无类别` 时
  - `/米家详情 [设备别名]` 将直接返回该设备支持的原始英文属性列表
- 新增“模板外原始属性补充显示”
  - 对于已配置类别的设备
  - 若探测到模板未覆盖的属性，会在详情页中额外列出，方便排障与后续适配

#### 🔧 调整
- `/刷新米家`
  - 现在统一展示为：
    - `【插件别名】(米家云端名称) [状态] (DID)`
  - 让列表展示更贴近米家 App 中的真实命名习惯
- `/米家详情`
  - 已配置有效类别时：
    - 优先展示中文化的“可调属性 / 状态传感”
    - 再尝试精准读取实时状态
  - 未配置类别时：
    - 不再硬套中文模板
    - 改为直接输出设备原始能力菜单
- `/米家帮助`
  - 现在优先根据你配置的设备类别输出更准确的帮助内容
  - 若设备为 `无类别`，则返回通用帮助，并提示可透传英文属性
- `/米家控制`
  - 控制格式保持不变：
    - `/米家控制 [设备别名] [属性] [值]`
  - 若设备已配置类别，则优先使用对应中文映射
  - 若设备未分类，仍支持直接透传原始英文属性名

#### 🛠 优化
- 优化了“别名 / 云端名称 / 控制模板”三者之间的职责分离：
  - 别名：用于机器人内控制入口
  - 云端名称：用于用户界面展示
  - 设备类别：用于中文模板匹配与帮助输出
- 降低了对设备 `name` 和 `model` 自动猜测的依赖，改为以用户手工配置类别为主
- 详情页在实时数据读取失败时，不再只返回“无数据”，而是尽量回退展示原始能力项

#### 🧯 修复
- 修复了设备在米家 App 中被改名后，详情页无法稳定命中原有中文模板的问题
- 修复了部分情况下 `/米家详情` 只剩默认开关信息、缺少有效排障线索的问题
- 改善了未适配设备的可观测性，避免“设备在线但什么都看不到”的体验

#### ⚠️ 配置迁移说明
本版本建议在保留原有 `device_map` 的基础上，新增配置 `device_category_map`。例如：

```json
{
  "空调": "空调类别",
  "净化器": "净化器类别",
  "风扇": "风扇类别",
  "蒸煮锅": "蒸煮锅类别",
  "空气炸锅": "空气炸锅类别",
  "手环": "无类别"
}
```

## [v6.3.11 ] - 2026-03-11

### ✨ 新特性 (New Features)

* **🚀 引入“千机千面”物模型架构 (Device Profiles)**
  * 彻底解耦“用户自定义别名”与“底层设备控制逻辑”。
  * 新增 `device_profiles.py` 独立配置库，通过官方名称特征词（如“空气净化器”、“落地扇”）自动挂载专属控制字典。
  * 轻松实现跨设备品类的自然语言控制（如：将人类语言的“睡眠”、“制冷”精准转译为底层所需的数字枚举）。
* **📊 全中文实况状态透传**
  * `/米家详情` 指令迎来史诗级视觉增强。
  * 自动嗅探设备公开属性，并内置黑名单过滤枯燥的工程级调试代码（如 `fault`, `dbg`, `crc32`）。
  * 将底层的英文传感器标识（如 `relative_humidity`, `filter_left_time`）自动翻译为优雅的全中文排版，并智能追加 `%`, `°C`, `天` 等单位。
* **🛠️ 开放式探索与共建引导**
  * 在 README 中加入标准的 Issue 提交模板，鼓励用户在使用 `/米家详情` 遇到未汉化的高级属性时提交数据，共建物模型图纸库。

### 🐛 修复 (Bug Fixes)

* 修复了并发修改持久化数据时的竞态风险（引入 `threading.RLock` 保护 `update_state`）。
* 修复了子进程退出和插件热重载 `terminate()` 时，底层会话生命周期管理混乱导致的悬空指针问题（缩小锁范围，仅用于引用交接）。
* 修复了指令前导空格处理不当导致的正则失效问题。
  
###  碎碎念 
* 让GPT优化了一下，代码已经完全看不懂也看不动了，测试下来是正常的就行。

## 🗺️ 未来更新计划 (Roadmap)


### 🌈 场景与自动化 (Smart Scenes)
* **原生/本地场景支持**：支持调用米家 App 预设场景或在插件内自定义快捷指令组（如 `/米家场景 晚安`）。

### 🛡️ 体验与稳定性增强 (UX & Stability)
* **状态快照缓存**：即使设备离线，也能显示最后一次成功读取的状态信息（如“5分钟前湿度为 60%”）。
* **控制回读验证**：下发指令后自动异步回读设备状态，确保控制真正生效。
* **更友好的错误解释器**：将枯燥的系统错误码转译为人类易懂的排障建议。

### ⌨️ 管理与效率工具 (Management Tools)
* **别名管理命令化**：脱离 WebUI，直接通过指令 `/米家绑定`、`/米家解绑` 管理设备别名。
* **档案一键导出**：支持导出当前设备的完整物模型 JSON，方便开发者调试与用户反馈。

### 🧠 深度智能化 (Intelligence)
* **自然语言模糊控制**：无需严格遵循指令格式，支持 `@机器人 把空调调到26度` 这种更自然的交互方式。
* **定时任务调度**：支持相对时间或绝对时间的控制预约（如“30分钟后关闭净化器”）。

### 🛠️ 关于Tool：
* 这是一个需要长期规划的事情，可能会做自然语言调取温度计之类的的数据，但是处于安全性考虑，大电器不会接入的，请**自行fork**。
